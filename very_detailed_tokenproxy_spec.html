Tokenproxy Stage One

A small Rust proxy for OpenAI-compatible agent traffic

This file is the implementation authority for stage two. It defines the behavior, architecture, crate choices, failure rules, data paths, and benchmark plan for coding tokenproxy.

Evidence snapshot: local checkout at /Users/vic8or/dev/tokenproxy, observed through 2026-05-27 00:14:12 PDT and 2026-05-27 07:14:12 UTC.

Abstract

Tokenproxy should be a deliberately small Rust proxy for OpenAI Chat Completions, OpenAI Responses, and Responses WebSocket traffic. Its job is to accept OpenAI-compatible agent requests, choose a usable upstream account, preserve streaming semantics, and keep the continuation path fast. It is not a gateway platform, an API translator for every provider, or a general QUIC experiment.

The central design claim is that the largest latency wins for Codex-like workflows come from keeping the model continuation path short: persistent Responses WebSocket connections, correct previous_response_id usage, prompt-cache-aware request construction, and compact state replay. Rust matters because the proxy should add negligible local overhead and predictable tail latency. Rust alone does not remove upstream model latency, DNS variance, cache misses, account throttling, or OpenAI incidents.

Claims Defended

Claim 1. The first production implementation should use Tokio plus Hyper/Axum-style HTTP primitives, a pooled Reqwest upstream HTTP client, and Tokio Tungstenite for WebSocket legs. It should not start with Pingora, io_uring, or first-party QUIC/H3 code.

Claim 2. The proxy should route by account health, model support, endpoint, and continuation state. It must not retry committed streaming output invisibly. This follows the proxy failure boundary described in Pingora: before downstream response headers, retry or fail over; after headers, log and stop because the client has already seen a response boundary.

Claim 3. Responses WebSocket support is not optional for agentic traffic. OpenAI documents WebSocket mode as a persistent /v1/responses connection that sends repeated response.create events with only new input plus previous_response_id. OpenAI also documents a single in-flight response per socket, no multiplexing, a 60-minute connection limit, and recovery rules for previous_response_not_found (OpenAI, 2026b).

Claim 4. Tokenproxy should preserve Responses output items exactly enough for stateless replay, including assistant phase when present, and should prefer previous_response_id when the upstream supports it. OpenAI's GPT-5.5 guidance requires preserving returned output items for stateless/ZDR flows and preserving phase on assistant output items when manually replaying state (OpenAI, 2026c).

Claim 5. Network path facts are observations, not constants. In this checkout, DNS resolved api.openai.com to 172.66.0.243 and 162.159.140.245. It resolved status.openai.com through cname.vercel-dns.com to 76.76.21.241 and 66.33.60.193. These values are inputs to the measurement log, not implementation constants.

Scope

Stage two should implement one binary named tokenproxy. The public API surface is:

  • POST /v1/chat/completions with JSON and SSE streaming support.
  • POST /v1/responses with JSON and SSE streaming support.
  • POST /v1/responses/compact as a thin pass-through for compacting Responses input windows.
  • GET /v1/responses WebSocket upgrade for Responses WebSocket mode.
  • GET /healthz for local process health.
  • GET /metrics for Prometheus-format counters and histograms.

The implementation must not expose general proxying. Requests outside the supported route set return 403 or 404 with a small JSON error. This mirrors the strictness in Codex's local Responses proxy, which accepts exactly POST /v1/responses, overrides upstream auth and Host, and rejects other requests (Codex, 2026a, lines 31-64; Codex, 2026b, lines 170-178).

Tokenproxy is allowed to run benchmark probes and diagnostic commands. It is not allowed to implement unrelated provider APIs, a dashboard, a plugin framework, or a storage engine beyond the account and health state required by routing.

OpenAI API Surface

The official endpoint list advertises https://api.openai.com/v1 as the base URL and includes /chat/completions, /responses, /responses/compact, /responses/input_tokens, /responses/{response_id}, /responses/{response_id}/cancel, and /responses/{response_id}/input_items among many other paths (OpenAI, 2026a). Tokenproxy stage two implements the two generation endpoints, /responses/compact, and the WebSocket transport path. It does not implement Conversations, Threads, Assistants, Batches, Realtime, Files, or Images.

POST /v1/responses returns either a JSON Response or a text/event-stream sequence of typed ResponseStreamEvent events. The response object includes continuation and orchestration fields that tokenproxy must preserve or surface: previous_response_id, reasoning, store, text, tool_choice, tools, truncation, usage.input_tokens_details.cached_tokens, and usage.output_tokens_details.reasoning_tokens (OpenAI, 2026a).

POST /v1/chat/completions remains supported because clients still use it. The OpenAPI summary recommends Responses for new projects and states that Chat Completions can return either a JSON chat completion object or streamed chat completion chunks (OpenAI, 2026a). Tokenproxy should pass Chat Completions requests through instead of converting them unless a benchmarked client workflow requires conversion.

Normative Stage-two Contract

This section is the implementation contract. Stage two is complete only when these behaviors are implemented and tested. The words must, must not, and should are normative.

Process contract

  • The binary name must be tokenproxy.
  • The default bind address must be 127.0.0.1:8787.
  • Binding to a non-loopback address must require server.allow_non_loopback = true.
  • Startup must fail before listening when the config file is invalid, no account is enabled, or an enabled account references a missing token environment variable.
  • Startup must not make credentialed OpenAI requests unless the operator explicitly runs a probe command.
  • Shutdown must stop accepting new sockets, let active HTTP/SSE/WebSocket sessions drain for shutdown_grace_ms, then close remaining sessions with metrics.
  • All timestamps in logs and benchmark artifacts must include timezone offset and UTC.

Route contract

Route Accepted request Proxy behavior Unsupported behavior
POST /v1/chat/completions JSON body. stream may be true or false. Forward opaque JSON. Classify model and stream flag. Do not convert to Responses. Wrong method returns 404. Non-JSON body returns 400.
POST /v1/responses JSON body. stream may be true or false. Forward opaque JSON. Parse routing fields. Capture output items from JSON and SSE when available. Wrong method returns 404. Non-JSON body returns 400.
POST /v1/responses/compact JSON body. Forward as an HTTP-only Responses route. Treat output as opaque compacted state. WebSocket attempts return 426 or a WebSocket close before upstream dial.
GET /v1/responses HTTP Upgrade to WebSocket. Accept JSON text frames for Responses WebSocket mode. Open one upstream socket per downstream session. Non-upgrade GET returns 426. Binary frames close with WebSocket code 1003.
GET /healthz No body. Return process liveness, version, and enabled account count. Do not expose secrets. Non-GET returns 405.
GET /metrics No body. Return Prometheus text exposition. Non-GET returns 405.

Header and body rules

  • Strip incoming Authorization and Host; insert selected account auth and upstream host per request.
  • Strip hop-by-hop headers: Connection, Keep-Alive, Proxy-Authenticate, Proxy-Authorization, TE, Trailer, Transfer-Encoding, and Upgrade.
  • Preserve upstream correlation and rate headers when valid HTTP header values: openai-request-id, x-request-id, cf-ray, retry-after, and x-ratelimit-*.
  • Do not buffer a streaming upstream body after downstream SSE headers are committed. Forward frames as they arrive.
  • Reject compressed request bodies with 415 in stage two. Supporting decompression requires separate tests for body limits, routing-field parsing, and byte-for-byte replay safety.
  • Hash body dumps with SHA-256 before logging. Raw body dumps are permitted only when request_body_dumps = true and must still redact configured JSON pointer paths.

Anatomy of the Proxy

Tokenproxy has five hot-path components. Keep each component small enough to benchmark in isolation.

1. Listener and request classifier

The listener accepts HTTP/1.1 and HTTP/2 client traffic on a local interface or configured bind address. The request classifier matches method and path, rejects unsupported routes before reading large bodies, and classifies the request as chat_json, chat_sse, responses_json, responses_sse, or responses_ws.

The classifier reads no more JSON than needed: method, path, body size, model, stream, store, previous_response_id, prompt_cache_key, service_tier, reasoning.effort, and text.verbosity. Use serde_json for stage two. Consider simd-json only after a local parse benchmark shows body parsing is a top contributor to proxy overhead.

2. Account router

The router chooses an upstream account by endpoint, model, health, backoff, and continuation constraints. Every account record must include:

  • id: stable local account identifier.
  • kind: openai_api_key or later ChatGPT-account adapter kind.
  • base_url: default https://api.openai.com/v1.
  • models: explicit allowlist or discovered model map.
  • supports_responses, supports_chat_completions, supports_responses_ws.
  • health: open, throttled, auth_failed, disabled, or unknown.
  • next_retry_at: monotonic deadline for retrying throttled accounts.
  • prompt_cache_key_seed: stable per caller or per account key material.

The router must never put secret-bearing default headers on a reusable client object. Insert account auth per request so pooled connections cannot carry headers across requests by construction. Codex's local proxy uses an in-memory auth header, marks it sensitive, and overrides Authorization and Host on each forwarded request (Codex, 2026b, lines 196-228).

3. HTTP upstream client

Use one shared Reqwest client per upstream origin and TLS policy. Enable connection pooling. Do not set a global read timeout for streaming responses. Codex's local proxy disables Reqwest's default 30-second timeout so long-lived response streams continue flowing (Codex, 2026b, lines 102-107).

Pingora's pooling documentation gives the correctness rule to copy: reuse only connections to the exact same peer identity, including IP:port, scheme, SNI, certificate verification settings, and proxy settings; mark errored connections non-reusable (Cloudflare, 2026b, lines 1-22). Reqwest handles the pool internally, but tokenproxy must keep client instances separated by upstream origin and TLS/proxy configuration.

4. Streaming relay

For SSE, tokenproxy forwards events as bytes and records response lifecycle events by parsing event frames opportunistically. OpenAI Responses streaming uses typed semantic events such as response.created, response.output_text.delta, response.completed, and error (OpenAI, 2026e). CLIProxyAPI already has practical repair code for malformed or incomplete Responses SSE output: it buffers frames, records response.output_item.done, and repairs response.completed.response.output when the completed payload lacks output items (CLIProxyAPI, 2026b, lines 48-119; CLIProxyAPI, 2026c, lines 160-220). Tokenproxy should implement the same behavior behind a small tested module named sse_repair.

The stream relay must delay committing downstream SSE headers until the first upstream data chunk or terminal upstream error. CLIProxyAPI does this by peeking at the first chunk and returning JSON errors before SSE headers are committed (CLIProxyAPI, 2026a, lines 450-524).

5. WebSocket session manager

A downstream Responses WebSocket session maps to one upstream Responses WebSocket session at a time. Do not multiplex unrelated downstream sessions over one upstream socket. OpenAI documents one in-flight response per socket and no multiplexing, so parallel runs need multiple connections (OpenAI, 2026b).

CLIProxyAPI's Responses WebSocket handler is the closest local reference. It accepts response.create and response.append, retains tool caches, pins upstream auth when incremental input is supported, carries lastRequest and lastResponseOutput, handles generate:false prewarm locally when needed, and falls back to transcript replay when the upstream cannot consume previous_response_id incremental input (CLIProxyAPI, 2026d, lines 208-280; CLIProxyAPI, 2026e, lines 306-429; CLIProxyAPI, 2026f, lines 458-613).

Why not Pingora first

Pingora is a strong reference for pooling, health checks, failover, and graceful upgrade. It also brings a larger proxy framework. Stage two needs OpenAI-specific JSON, SSE, and WebSocket continuation logic more than a general reverse-proxy stack. Use Pingora's rules, not its framework, until a benchmark shows the Hyper/Axum implementation misses overhead targets.

Why not io_uring first

Monoio shows the performance appeal of thread-per-core and io_uring for network-bound servers, but it also documents platform constraints, runtime configuration requirements, unstable features, and compatibility costs (Bytedance, 2026a, lines 21-31; Bytedance, 2026b, lines 63-87). Tokenproxy's upstream work is TLS, HTTP/2, SSE, and WebSocket heavy; Tokio support is better across the chosen libraries. Use io_uring only after a local Tokio implementation has a measured kernel or scheduler bottleneck.

Why not QUIC/H3 first

Quiche and s2n-quic show that QUIC/H3 is powerful and complex. Quiche provides a low-level QUIC packet API and makes the application provide sockets, an event loop, and timers (Cloudflare, 2026d, lines 8-11). It also requires QUIC flow control, ALPN, TLS configuration, and BoringSSL build machinery (Cloudflare, 2026d, lines 81-103; Cloudflare, 2026e, lines 330-343). s2n-quic is extensively tested and provider-driven, but it requires QUIC-specific provider choices and has GSO requirements on older Linux kernels (AWS, 2026a, lines 3-10, 116-118). Tokenproxy should measure HTTP/3 with curl and external probes first. It should not implement custom H3 until repeated measurements from target networks show a durable advantage for OpenAI endpoints.

Data Paths

Non-streaming HTTP

  1. Classify route and read body with a configured max size.
  2. Parse routing fields only.
  3. Select an account whose health permits the endpoint and model.
  4. Build upstream URL by appending the exact route to the account base URL.
  5. Forward request headers except hop-by-hop headers and incoming Authorization.
  6. Insert account Authorization and Host for the upstream origin.
  7. Send with pooled Reqwest client.
  8. Mirror upstream status, content type, request ID headers, and body.

SSE HTTP

  1. Perform the non-streaming steps through upstream send.
  2. Read the first upstream event or upstream terminal error before committing downstream SSE headers.
  3. If upstream failed before event bytes, return JSON error and mark account health.
  4. If upstream produced event bytes, commit SSE headers and forward frames.
  5. Parse Responses event types for metrics and output replay state.
  6. On client disconnect, cancel upstream request and record cancellation.

Responses WebSocket

  1. Accept downstream WebSocket only on /v1/responses.
  2. Require client messages to be JSON text with type=response.create.
  3. On first request, select an account supporting Responses and, when requested, WebSocket continuation.
  4. Open upstream wss://api.openai.com/v1/responses or account base equivalent.
  5. Forward response.create unchanged except for safe defaults and prohibited transport fields.
  6. For later turns, prefer previous_response_id plus only new input when upstream state is valid.
  7. If upstream lacks WebSocket continuation, synthesize a full input array from last request, last output, and new input.
  8. Forward upstream JSON text events to the downstream in order.
  9. Capture response.completed.response.id, output items, and error codes.
  10. Close upstream and downstream on either leg closing; do not hide mid-stream failures.

Header and Body Contract

Header handling must be mechanical and covered by unit tests. Codex's local Responses proxy provides the model: forward ordinary request headers, replace incoming Authorization, replace Host, and mark the upstream auth header as sensitive (Codex, 2026b, lines 196-228).

Inbound request handling

  • Reject request bodies larger than server.max_body_bytes before selecting an account.
  • For JSON routes, read the body once into bytes, parse routing fields from those bytes, then forward the original bytes unless a tested transform applies.
  • For WebSocket, accept only text frames containing a JSON object with type = "response.create".
  • Generate a tokenproxy_request_id for every accepted request and include it in logs and downstream error responses.
  • Do not log raw request bodies unless observability.request_body_dumps is enabled.

Forwarded request headers

Header class Rule
Authorization Always strip inbound value. Insert Bearer <selected account token> per request or per WebSocket handshake.
Host Always strip inbound value. Insert host from selected account base_url.
Hop-by-hop headers Strip connection, keep-alive, proxy-authenticate, proxy-authorization, te, trailer, transfer-encoding, and upgrade except during WebSocket handshake construction.
Content length Let the HTTP client set it from the forwarded body bytes.
Content type and accept Forward unless invalid. Preserve text/event-stream semantics for streaming responses.
OpenAI request headers Forward safe client-supplied OpenAI headers, including organization/project headers, when explicitly allowed by config. Default deny for unknown OpenAI-* request headers until tests cover them.
Proxy diagnostics Add x-tokenproxy-request-id. Do not forward internal account IDs or health state upstream.

Response handling

  • Mirror upstream status and body for upstream responses.
  • Forward content-type, openai-request-id, x-request-id, and rate-limit headers.
  • Strip set-cookie unless a later ChatGPT-account adapter proves it is needed and stores it out of logs.
  • For SSE, do not synthesize semantic events except for tested repair of missing completed output when source events contain enough output-item data.
  • For WebSocket, forward upstream JSON text events in order. Do not reorder deltas for metrics convenience.

Account Selection Algorithm

Selection must be deterministic for identical state so tests and benchmark replays can reproduce routing decisions. Randomness is allowed only as a final tie-breaker when the seed is logged.

Eligibility filters

  1. Account is enabled.
  2. Account health is open, unknown, or throttled with now >= next_retry_at.
  3. Account supports the requested endpoint.
  4. Account model allowlist is empty or contains the requested model after alias normalization.
  5. Requested service tier is empty or listed in account service_tiers.
  6. WebSocket requests require supports_responses_ws = true.
  7. Continuation pinned to an account must use that account unless the failure policy explicitly permits failover and replay can remove stale previous_response_id.

Scoring

After filtering, sort by this tuple. Lower values win except where noted. 

(
  continuation_penalty,       // 0 for pinned compatible account, 10 otherwise
  health_penalty,             // open=0, unknown=20, cooled_down_throttled=40
  -account.priority,          // higher configured priority wins
  ewma_connect_ms_bucket,     // coarse bucket, not raw high-cardinality data
  ewma_first_event_ms_bucket, // coarse bucket per endpoint/model family
  recent_failure_count,
  stable_hash(account_id, caller_hash, model_family)
)

The router must log the winning account hash and the first excluded reason for every losing account at debug level. Metrics must count exclusions by reason without exposing raw account IDs.

Health transitions

Observed event Transition Reset rule
Successful response headers or WebSocket event unknown or throttled to open Clear transient failure count for that endpoint/model family.
401 or 403 from upstream auth_failed Only manual reload or changed token can reopen the account.
429 with Retry-After throttled until parsed retry deadline First success after deadline returns to open.
429 without Retry-After throttled with exponential jittered backoff Backoff resets after success or five quiet minutes.
Connect, DNS, or TLS failure Transient failure count increments Account remains eligible after backoff unless failures exceed configured threshold.
previous_response_not_found No account health change by default Mark replay state invalid for that session and retry only with full replay when complete.

State Machines

The implementation should use explicit enums for these states. That makes logs readable and keeps retry boundaries testable.

HTTP JSON state machine

Accepted
  -> BodyRead
  -> Classified
  -> AccountSelected
  -> UpstreamSent
  -> DownstreamCommitted
  -> Completed | ClientCancelled | UpstreamFailed

Retry is legal only before DownstreamCommitted and only when the original body bytes are still available. Because stage-two HTTP JSON bodies are read into bytes before forwarding, a single retry before commitment is mechanically possible, but it must still obey endpoint and account policy.

SSE state machine

Accepted
  -> BodyRead
  -> Classified
  -> AccountSelected
  -> UpstreamSent
  -> WaitingFirstFrame
  -> DownstreamCommitted
  -> Relaying
  -> Completed | ClientCancelled | UpstreamStreamFailed

WaitingFirstFrame is the safety boundary. If upstream returns an HTTP error before the first SSE frame, tokenproxy may return a JSON error or retry once. After the first frame reaches the downstream client, tokenproxy must not hide failures by opening a second upstream stream.

WebSocket state machine

Accepted
  -> WaitingClientCreate
  -> AccountSelected
  -> UpstreamConnecting
  -> UpstreamReady
  -> InFlight
  -> UpdatingReplayState
  -> WaitingClientCreate
  -> Closing

A WebSocket session has exactly one in-flight response. If the downstream client sends a second response.create while the session is in InFlight, tokenproxy must emit a websocket_in_flight error frame and must not forward the second request upstream. OpenAI documents the same single in-flight rule for Responses WebSocket mode (OpenAI, 2026b).

WebSocket session record

struct WsSessionState {
  session_id: String,
  account_id_hash: String,
  upstream_origin: String,
  supports_incremental_previous_response_id: bool,
  supports_compaction_replay: bool,
  in_flight: bool,
  last_request_template: serde_json::Value,
  last_completed_response_id: Option<String>,
  last_completed_output_items: Vec<serde_json::Value>,
  replay_generation: u64,
  last_activity_at: Instant,
}

last_completed_output_items must preserve each item as opaque JSON. Do not deserialize into a lossy enum unless every unknown field is retained. This is required for assistant phase preservation and future output item shapes (OpenAI, 2026c).

Responses Orchestration Rules

Feature Proxy rule Evidence
previous_response_id Preserve it for HTTP and WebSocket. In WebSocket mode, continue from the most recent completed response on that socket. If an uncached ID fails, replay full context when tokenproxy has it; otherwise return the upstream error unchanged. OpenAI documents connection-local most-recent response cache, store=false behavior, eviction after failed turns, and recovery with full context (OpenAI, 2026b; OpenAI, 2026g).
Output item replay Store completed output items in session memory until the session closes or the retention cap expires. Use them only for stateless replay or fallback. OpenAI recommends passing returned output items each turn for stateless/ZDR flows (OpenAI, 2026c). CLIProxyAPI repairs missing completed output from streamed output-item events (CLIProxyAPI, 2026c).
Assistant phase Never drop phase from assistant items when replaying output items. Treat output items as opaque JSON unless a tested transform requires a field. OpenAI GPT-5.5 guidance says manually managed Responses state must preserve assistant phase unchanged (OpenAI, 2026c).
Prompt caching Keep stable prompt/tool prefixes first, dynamic items last, pass through prompt_cache_key, and log cached token counts. For caller-specific cache routing, derive a stable key from caller identity and model family. OpenAI documents exact prefix matching, 1024-token eligibility, prompt_cache_key, cached token logging, and cache overflow around high request rates for the same prefix/key (OpenAI, 2026d). CLIProxyAPI tests a stable key derived from user API key and uses it as Session_id (CLIProxyAPI, 2026g, lines 16-64).
Reasoning effort Pass through reasoning.effort. Default decisions belong to clients, not tokenproxy. Metrics must bucket by effort because latency and quality shift with effort. OpenAI documents low, medium, high, and xhigh choices, with medium as GPT-5.5 default and low recommended for many latency-sensitive workflows (OpenAI, 2026c).
Verbosity Pass through text.verbosity and include it in benchmark labels. OpenAI documents verbosity as separate from reasoning quality and recommends intentional use for concise responses (OpenAI, 2026c).
Tool preambles Preserve preamble output items and assistant state. Do not filter them from streams or replay. OpenAI says preambles improve UX and allow continuation from the same assistant state after tool results arrive (OpenAI, 2026c).
Compaction For server-side compaction, forward context_management and continue with normal replay rules. For standalone /responses/compact, proxy the HTTP call as a Responses endpoint, treat the returned compacted window as opaque JSON, and start a new WebSocket chain when that compacted window is used. OpenAI documents server-side compaction via context_management, a stateless standalone compact endpoint, and the rule to pass standalone compact output as-is (OpenAI, 2026h).

Replay and WebSocket Wire Contract

Replay code is the highest-risk JSON transform in tokenproxy. It must live in one module, have golden tests, and preserve unknown JSON fields by default.

Accepted WebSocket messages

Client frame Accepted Action
Text JSON object with type = "response.create" Yes Normalize, route, and forward as a Responses request event.
Text JSON object without type No Return a WebSocket error frame with websocket_unsupported_message.
response.append No for tokenproxy stage two Return a WebSocket error frame. Codex's current executor forces every upstream WebSocket request to response.create with previous_response_id for incremental turns (CLIProxyAPI, 2026h, lines 678-687).
Binary frame No Close with WebSocket code 1003.
Ping Yes Reply with Pong without touching replay state.

First turn

The first accepted WebSocket response.create must include a model. Tokenproxy forwards it as JSON text to the upstream socket after account selection and auth insertion. It records a request template after removing transport-only fields and after applying any tested defaults. It records last_completed_response_id and last_completed_output_items only after a terminal response.completed event. 

{
  "type": "response.create",
  "model": "gpt-5.5",
  "instructions": "You are a concise coding agent.",

Incremental fast path

The fast path is allowed only when the session has a completed response ID, the selected account supports WebSocket incremental input, and no prior event invalidated connection-local state. The forwarded request keeps previous_response_id and includes only new input items. OpenAI documents this pattern for long-running tool-call workflows (OpenAI, 2026g). 

{
  "type": "response.create",
  "previous_response_id": "resp_abc123",
  "input": [{
    "type": "function_call_output",
    "call_id": "call_1",
    "output": "{\"status\":\"ok\"}"
  }]
}

Full replay fallback

Full replay is required when the upstream account cannot use incremental previous_response_id, when a connection-local previous response is unavailable, or when failover switches to a different account. The transform must apply these steps in order:

  1. Start from last_request_template.
  2. Remove previous_response_id.
  3. Copy missing stable top-level fields from the template: model, instructions, tools, tool_choice, parallel_tool_calls, reasoning, text, truncation, store, and service_tier.
  4. Build input as prior input items, then completed output items, then new input items.
  5. Preserve output items exactly, including phase, tool-call IDs, annotations, reasoning summaries, and unknown fields.
  6. Deduplicate tool-call output items by call_id only when a test proves the same tool output appears in both prior and new input.
  7. Set stream = true for WebSocket relay requests.
{
  "type": "response.create",
  "model": "gpt-5.5",
  "instructions": "You are a concise coding agent.",
  "tools": [{ "type": "function", "name": "run_test", "parameters": { "type": "object" } }],
  "input": [
    { "type": "message", "role": "user", "content": "Run the fast tests." },
    { "type": "function_call", "call_id": "call_1", "name": "run_test", "arguments": "{\"target\":\"fast\"}" },
    { "type": "function_call_output", "call_id": "call_1", "output": "{\"status\":\"ok\"}" }
  ],
  "stream": true
}

Compaction reset

After /v1/responses/compact returns a compacted input window, the next WebSocket request using that compacted window must start a new replay generation. Tokenproxy must not prepend stale pre-compaction input or stale output items to that request. The state transition is: 

Before compaction:
  replay_generation = 7
  last_completed_response_id = Some("resp_old")
  last_completed_output_items = [...]

After compacted window is used:
  replay_generation = 8
  last_completed_response_id = None until next completion
  last_completed_output_items = []
  last_request_template = compacted request window

Failure Behavior

Failure handling must be deterministic. Hidden retries are allowed only before downstream commitment and only when the upstream request has not sent a body or when replay is explicitly safe.

Failure Action Health update
DNS/connect/TLS failure before upstream body Try another eligible account once. Return 502 if none. Mark transient failure; short exponential backoff.
HTTP 401 or 403 Return upstream error. Do not retry another account unless client opted into failover for auth errors. Mark account auth_failed.
HTTP 429 Return upstream error unless a retry-before-commit policy is enabled and body replay is safe. Honor Retry-After when present. Otherwise back off with jitter.
HTTP 500 or 503 before stream commit Retry another eligible account at most once. Preserve original request ID in logs. Short backoff. Check status page during operator investigation.
SSE error after first downstream event Forward error event or close stream. Do not retry invisibly. Mark streaming failure with model and account labels.
WebSocket previous_response_not_found Replay full context once if local session state is complete. Otherwise forward error. Do not mark account unhealthy unless repeated outside reconnects.
WebSocket 60-minute limit Open a new upstream socket and continue with persisted ID or full replay. None.
Client disconnect Cancel upstream HTTP request or close upstream WebSocket. None.

The non-idempotent retry rule comes from Pingora's failure guide: retrying a POST after the request has already been sent is unsafe unless the upstream semantics make it safe; after downstream headers are sent, the proxy can only log and give up (Cloudflare, 2026c, lines 3-17).

OpenAI error guidance requires backoff and rate respect for 429, a brief wait and status-page checks for 500/503, gradual ramp after Slow Down, and special handling for previous_response_not_found and websocket_connection_limit_reached (OpenAI, 2026f).

Benchmark Methodology

Benchmarks must be workflow benchmarks, not microbenchmarks alone. Ripgrep's README warns that a single benchmark is never enough and shows performance cliffs where output volume or pattern shape dominates tool differences (BurntSushi, 2026a, lines 47-49; BurntSushi, 2026b, lines 87-108). Uv's benchmark notes separate warm and cold cases and warn that OS, filesystem, package set, and network behavior can change results (Astral, 2026, lines 3-13, 18-48). Tokenproxy inherits the same rule: label the environment, distinguish warm/cold connection and cache states, and report tails.

Required environment fields

  • Timestamp with timezone and UTC.
  • Host OS, CPU, RAM, kernel, network type, and region.
  • Tokenproxy git commit and build profile.
  • Submodule commit snapshot when source comparisons are used.
  • curl version, Rust version, OpenSSL/rustls version, and DNS resolver.
  • Resolved IPs, TLS version, ALPN, connect time, TLS time, TTFB, total time.
  • Model, endpoint, transport, service tier, reasoning effort, verbosity, store flag.
  • Prompt-cache fields: key, retention, cached token count, cache hit ratio.

Required workflow cases

Single non-streaming Chat Completions request Route parse, account selection, upstream round trip, JSON body relay. Local proxy overhead p50 below 1 ms, p99 below 5 ms excluding upstream time.
Single Responses SSE request SSE header delay, first event latency, frame relay overhead, output item capture. First byte overhead below 1 ms p50 and no event reordering.
Codex 25 tool-call loop over HTTP Responses Repeated request overhead, prompt cache behavior, output replay size growth. Record TTFT and total time per turn; compare against direct client baseline.

State and Concurrency

The hot path should be read-mostly and allocation-conscious.

  • Use an immutable account routing table behind ArcSwap.
  • Use per-account health cells with atomic state and Instant deadlines.
  • Use bounded channels for WebSocket session events; on overflow, close the session rather than growing unbounded memory.
  • Store WebSocket replay state per session: last request JSON, last response output items, last response ID, account ID, and a compaction flag.
  • Expire closed session replay state immediately unless debug capture is explicitly enabled.
  • Never store raw Authorization in logs, metrics, or debug dumps.

LiveKit's data-channel sender is a useful reference for backpressure: it uses a low-buffer threshold, ordered in-flight packets, and a tiny pending queue so old data does not grow without bound (LiveKit, 2026b, lines 108-146, 177-228). Iggy's shard router gives the corresponding server-side rule: never block the reactor on a full internal channel; record the drop and rely on the protocol recovery path where one exists (Apache Iggy, 2026d, lines 178-227).

Security and Secret Handling

  • Bind to loopback by default. Non-loopback bind requires an explicit config flag.
  • Reject unsupported routes. Do not proxy arbitrary URLs.
  • Strip incoming Authorization before upstream forwarding.
  • Attach upstream Authorization only from selected account state.
  • Redact Authorization, cookies, Set-Cookie, account tokens, and refresh tokens in logs.
  • Expose account IDs in logs as stable hashes unless debug mode is enabled.
  • Use constant-time comparison for local admin tokens if an admin endpoint is added.
  • Disable request/response body dumps by default.
  • When dumps are enabled, write JSONL with redacted headers and a max body byte cap.

Codex's local proxy is strict because it is intended to protect an API key held by a privileged process from unprivileged callers. It reads the key from stdin, redacts sensitive dump headers, marks auth header values sensitive, and limits allowed requests (Codex, 2026a, lines 35-67; Codex, 2026a, lines 96-108). Tokenproxy has more account-routing behavior, but the same secret discipline.

Tradeoffs and Anti-pitch Notes

  • Rust cannot make model inference fast. It can keep proxy overhead predictable.
  • WebSocket mode helps long tool-call chains. It adds session state, reconnect logic, and fallback complexity.
  • previous_response_id reduces client payload size, but prior input tokens in the chain are still billed (OpenAI, 2026g).
  • Prompt caching can reduce latency, but cache hit rate depends on exact prefixes, routing, request rate, and model cache policy.
  • HTTP/3 support in curl does not prove OpenAI's path is faster over HTTP/3 from a target network.
  • Account failover improves availability before commitment. After stream commitment, transparent failover breaks correctness.
  • Typed request structs improve compile-time clarity but increase churn as OpenAI schemas evolve. Stage two should proxy opaque JSON and parse fields needed for routing.
  • Pingora, Monoio, quiche, and s2n-quic are valuable references. Their code size and domain scope exceed stage-two needs.
